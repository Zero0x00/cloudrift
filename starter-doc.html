What Cloudrift is and how to use this guide.

Welcome

Cloudrift is built around calling AWS APIs to inventory org accounts, DNS, edge services (S3 website, CloudFront, …), and IAM trust — then writing results to JSON folders on disk. A web dashboard in the same binary reads those files. We added demo generate so you can learn the UI and report formats without dialing AWS — that is training data, not a substitute for a real scan.

What this means

Production value comes from live AWS reads. Demo mode exists purely so designers, reviewers, and new contributors see populated tables and charts while the default scan path is still being wired to full collectors.

Who should use it: security engineers, cloud engineers, FinOps-adjacent folks, and leaders who want evidence-backed answers about forgotten DNS and external IAM trust.

Implemented works in code today · Skippable add-on (e.g. OpenAI key for embeddings) · Graph tier Neo4j-coupled advanced path · Stubbed registered but not wired · Planned on the roadmap

Orphaned edge and external trust in plain language.

The problem

Orphaned edge assets — A DNS name (like docs.example.com) still points at AWS (S3 website hosting, CloudFront, etc.) but the resource behind it is gone or mis-linked. Attackers sometimes reclaim the same bucket name or abuse a dangling distribution.

What this means

Your customers still hit the hostname in a browser. If an attacker controls the backing resource, they control what users see.

Stale external IAM trust — An IAM role in your account can be assumed by an external AWS account, SAML IdP, or OIDC app. If nobody reviewed it lately, it is easy to forget — but the trust stays valid until you remove it.

Why cost shows up — Even security issues have a waste angle: paying for DNS, certificates, or unused edge while also carrying takeover risk.

What Cloudrift looks for and why it matters.

Kinds of issues (and why they matter)

Findings fall into two big families: orphaned edge (public hostnames and AWS edge services) and external access (IAM trust to the outside world). Severity and evidence differ by subtype.

Orphaned edge — subdomain takeover and friends

S3 website takeover risk — DNS still points at an S3 website endpoint, but the bucket is gone or not in your org. An attacker who creates a bucket with the matching global name can host content under your hostname. Why it matters: phishing, malware distribution, and brand damage under a trusted domain.

Dangling CloudFront / CDN edge — The name resolves to AWS infrastructure, but the distribution, origin, or alternate-domain mapping is wrong. Responses may be 403/404 with recognizable AWS bodies. Why it matters: users still trust the URL; attackers probe these patterns at scale.

CDN hostname mismatch — Traffic hits CloudFront IPs but the requested Host is not on the distribution’s allowlist. Why it matters: subtle outages, possible routing confusion, and sometimes a path to serving unexpected content.

Broken DNS (NXDOMAIN / timeout) — The record is stale or misconfigured. Lower direct takeover pressure than a reclaimable S3 name, but still noise for security and IT — and it hides drift (“did we mean to delete this?”).

External IAM trust — cross-account and federated access

Ghost or admin-equivalent external trust — A role trusts an external account, SAML IdP, or OIDC app and carries broad permissions (or has never been used). Why it matters: durable lateral entry: whoever controls that external principal can assume into your estate.

Unapproved vendor account — Trust exists to an AWS account ID that is not on your allowlist. Why it matters: forgotten SaaS integrations and M&A leftovers are common sources of long-lived access.

Stale vs active trust — RoleLastUsed and policy heuristics separate “probably forgotten” from “still exercised.” Why it matters: prioritization — rotate or remove dead trust before it becomes an attacker’s path of least resistance.

For scoring tables and service coverage detail, see docs/security-coverage.md.

Four questions the product is built around.

What Cloudrift answers

  1. What exists? Hostnames, buckets, distributions, roles — tied to accounts when collectors have run.
  2. Who owns it? Account IDs, tags, and OU paths help teams route work.
  3. Is it claimable? For edge findings, claimability answers whether a stranger could plausibly take control.
  4. What is it costing? Static estimates and optional Cost Explorer merge (edge module) give FinOps-friendly signals.

Build, demo, dashboard, report.

Quick start (copy-paste)

Prerequisites: Go 1.24+, Node.js 20+ and npm (only if you build the embedded UI from source).

git clone https://github.com/Zero0x00/cloudrift.git && cd cloudrift
go mod download && go test ./...
cd dashboard && npm ci && npm run build && cd ..
go build -o cloudrift ./cmd/cloudrift

Or use make build from the repo Makefile (runs dashboard build + Go build).

Try without touching AWS

Use this block to visualize the product (dashboard, reports, optional graph export). It does not replace live AWS inventory — production findings come from real API calls once the scan pipeline is fully wired.

cloudrift demo generate --output-dir ./cloudrift-output
cloudrift report --scan-id latest --format table
cloudrift dashboard --output-dir ./cloudrift-output --open

Open http://127.0.0.1:8080 and pick the newest scan or use ?scan_id=demo if you generated into demo.

Keys, profiles, roles, and config.

AWS credentials explained

Access keys are long-lived ID/secret pairs. Profiles are named sections in ~/.aws/credentials and ~/.aws/config. The AWS SDK picks credentials using a default chain (env vars, profile files, SSO, etc.).

What this means

You do not type secrets into Cloudrift. You configure the machine the same way you configure the AWS CLI, then Cloudrift uses the SDK.

CLI today: set [aws].management_profile = "prod" in cloudrift.toml, or leave it empty to use the default chain (including AWS_PROFILE). There is no --profile flag on CLI commands.

[aws]
management_profile = "my-readonly-profile"

Multi-account: your management identity must be allowed to sts:AssumeRole into CloudriftAuditRole in each member account (see IAM page + docs/iam-setup.md). Access keys alone in one account do not magically see every account — the role pattern must exist.

What scan does today vs intended.

Real scan path

Intended: call AWS APIs across org accounts, validate DNS/HTTP, score risk, write findings.

Today: cloudrift scan still validates AWS credentials and creates a new timestamped folder under output_dir with scan-metadata.json and an empty findings.json. That proves the path and permissions; it is not yet a populated live assessment.

Scan IDs: the folder name (for example 20260514-120000). The special name latest means “newest by scan-metadata.json timestamp”.

Source of truth: JSON files on disk. The API never “fixes” findings in memory for long-term storage.

Every file under a scan directory.

Output artifacts

  • scan-metadata.json — one object with scan id, timestamps, counts, totals.
  • findings.json — array of findings; may be empty after default scan.
  • relationships.json — optional graph edges for export (POINTS_TO, TRUSTS, …).
  • assets/*.json — optional arrays of asset nodes; merged in lexical file name order.

There is no separate summary.json file — summaries are computed by the API from these inputs.

Pages and how data reaches the UI.

Dashboard guide

Start: cloudrift dashboard --output-dir ./cloudrift-output --port 8080 --open

The UI calls the same host’s /api/... routes. It lists scans by reading directories under output_dir. Main workflows (findings tables, diff, trust reports, accounts) use JSON only. Graph-tier pages (blast explorer, vector-backed query UI) need Neo4j + export — APIs return graph_available: false when the graph tier is absent.

Routes (high level)

  • /overview — executive / high-signal / operations modes.
  • /scan-control — start scan, pick AWS profile for API, live progress.
  • /findings, /triage — tables.
  • /accounts, /diff, /trust-report, /external-entities.
  • /blast-explorer — best with Neo4j + relationships.
  • /alerting — Slack webhook rules.
  • /query — UI for graph retrieval when configured.

Neo4j graph tier coupled to advanced features.

Neo4j & graph tier

Neo4j is not “a random extra database.” Cloudrift is coupled to it for an advanced tier of capability layered on the same JSON scans:

  • Graph relationships — structured POINTS_TO, TRUSTS, OWNED_BY, etc., for multi-hop reasoning.
  • Blast-radius exploration — UI and APIs that walk the graph from a finding, external entity, or principal.
  • Vector retrieval — embeddings over finding text for similarity search.
  • cloudrift query — CLI (and dashboard /query) over that stack; today retrieval-only, designed to grow toward richer RAG-style investigation.

Main workflows without Neo4j: everything that only needs findings.json / scan-metadata.json still works — the product deliberately degrades when the graph tier is off.

Technically, export re-reads JSON from disk and upserts into Neo4j; if you wipe Neo4j, your files on disk remain authoritative.

Enable with [neo4j] in TOML and --neo4j on scan or demo generate after files exist.

Vectors and retrieval-only CLI.

Embeddings & query

Embeddings are numeric vectors that capture “meaning” of text so Neo4j can rank similar findings. They power the graph tier query path together with Bolt.

Default provider: OpenAI text-embedding-3-small (384 dimensions). Set OPENAI_API_KEY (or your configured env name).

Local provider: Stubbed — selecting local returns a clear error until an on-box model ships.

cloudrift query: prints grounded retrieval results and explicitly states that answer synthesis is not implemented.

Audit role and least privilege.

IAM & permissions

Cloudrift’s org template defines a read role CloudriftAuditRole in member accounts, assumable from a management principal with an external ID condition.

The role is read-oriented (list/get across Route53, S3, CloudFront, IAM, Organizations, etc.) — not AdministratorAccess for Cloudrift itself.

See iam/stackset-template.yaml and docs/iam-setup.md for exact actions.

Practical triage loop.

Security engineer workflow

  1. Generate or import a scan with non-empty findings.json (demo is easiest).
  2. Use High-signal overview mode or API /top-fixes for ordering.
  3. Drill into a finding — read evidence fields; they are meant for humans.
  4. Use diff
  5. Graph tier: export to Neo4j when you need blast-radius, relationship-native views, and query / future RAG workflows.

Risk narrative without hype.

CISO / leadership view

  • Evidence first — each row ties to ARNs, hostnames, or trust principals.
  • Local control — artifacts stay on disk unless you opt into cloud APIs (OpenAI for embeddings only when you enable that path).
  • Honest scope — say plainly when a scan directory is empty; do not oversell “full org discovery” until orchestration lands.

Where to change code and tests.

Tests: go test ./...; dashboard: cd dashboard && npm ci && npm run test:run. After code edits: .venv-graphify/bin/graphify update . if you maintain the graph.

Common failures and fixes.

Troubleshooting

  • Credentials not found — verify aws sts get-caller-identity with the same profile / env as Cloudrift.
  • Access denied on AssumeRole — trust policy, external ID, and role name must match template.
  • Empty dashboard — check --output-dir; ensure findings.json is not empty (use demo generate).
  • Neo4j connection failed — Bolt URI, firewall, password_env variable set.
  • OpenAI / embeddings errors — key missing or wrong env name in TOML.
  • Local embeddings unavailable — expected until implemented; use OpenAI or skip query.
  • tsc: command not found — run npm ci in dashboard/ before npm run build.

Terms with short definitions.

Glossary

Scan
One run that produces a folder under output_dir.
Scan ID
Folder name (or latest alias).
Finding
One JSON object describing a risk with severity and evidence.
Asset
AWS or DNS object discovered by collectors.
Relationship
Directed edge between assets (e.g. DNS POINTS_TO bucket).
Claimability
How exploitable an edge hostname is (e.g. reclaimable vs dangling).
Dangling
Resolves to AWS but backing resource is broken or ambiguous.
Reclaimable
Attacker could plausibly register/recreate the target resource.
External principal
Non-internal identity trusted by an IAM role policy.
Neo4j (graph tier)
Coupled store for relationships, blast radius, vectors, and query; main JSON workflows still run without it.
Embedding
Vector representing text for similarity search.
RAG (here)
Retrieval-augmented lookup: fetch similar findings — no generated prose answers yet.
CloudriftAuditRole
Read-only member-account role assumed from management.
